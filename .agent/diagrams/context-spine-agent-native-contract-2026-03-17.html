Context Spine Contract

Native memory for the agent. Visual explanation for the human.

Context Spine should treat retrieval as a supported subsystem, not as the thing the memory surface fundamentally is. The agent should get a restart-safe working set and direct evidence. The human should get an honest visual explanation with links back to source truth. The whole system remains a project memory substrate, not a control plane.

Primary bootstrap + working set + evidence
Human visual-first output
Supported QMD / lexical / vector retrieval
Boundary memory substrate, not control plane
Agent contract

Open the repo, recover the working set, inspect local truth surfaces directly, and promote or invalidate memory deliberately when the model changes.

Human contract

Do not make the operator reason about indexes, embeddings, or internal records if a visual explainer and evidence trail can communicate the same thing more cleanly.

Design warning

If vector hydration or one search backend goes red, agent memory quality should degrade gracefully instead of collapsing.

Primary Agent Path

The memory path should survive without a perfect retrieval backend

This is the sequence the agent should rely on first. Retrieval helps when discovery is the bottleneck, but the core path is bootstrap to evidence to explicit reconciliation.

01

Bootstrap

Use context:bootstrap to recover the baseline, hot memory, latest session, and reading order.

02

Working Set

Read the baseline, current session, hot-memory index, and the explicit source_of_truth files.

03

Direct Evidence

Prefer code, docs, tests, git state, and command output when those surfaces are already local and bounded.

04

Explain

Return a visual explanation, evidence trail, and source references so a human can verify the model without decoding internals.

05

Reconcile

Use context:promote or context:invalidate when durable memory needs to change. Do not leave new truth trapped in runtime state.

Primary Versus Secondary

What defines the contract, and what remains a supported retrieval layer

Core memory

Agent-native surfaces

  • Baseline notes define the stable project reading path.
  • Session notes preserve short-horizon state and verification.
  • Hot memory compresses the current working set.
  • ADRs, runbooks, and diagrams keep durable truth inspectable.
  • Structured records support query, promotion, and reconciliation.
Support layer

Supported retrieval layer

  • QMD remains a supported retrieval surface for broad discovery and human lookup.
  • Lexical search keeps the system usable when vectors are unavailable.
  • Vector search improves recall but is not the definition of the memory surface.
  • Embeddings should improve the experience, not gate the basic contract.
When To Reach For Retrieval

Use retrieval when discovery is the expensive part

Good fit

Cold start or handoff

Many candidate notes, multiple repos, and low confidence about where the relevant thread lives.

Good fit

Broad corpus sweep

Questions that span notes, docs, skills, or external durable memory where brute-force direct reads would waste time.

Not the first move

Bounded live repo work

When the file set is already small and the authoritative code/docs/tests are in hand, direct reading wins on trust and latency.
