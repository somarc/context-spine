Context Spine · Doctor

Audit the memory spine before drift turns into confusion.

context:doctor is the hygiene pass that checks whether Context Spine still maps cleanly to real project truth. It is not about "more notes." It is about whether the baseline, sessions, docs, generated aids, and explainers still point at the right things.

What it catches
Silent drift Stale sessions, broken source paths, missing docs boundaries, or explainers with no durable home.
Run it after major doc changes, before phase closeout, before handoff, or whenever the repo feels harder to trust than it should.
Doctor Loop
1. Read the baseline contract

Verify that a baseline note exists and that its source_of_truth still resolves to live files.

2. Compare work against memory

Check whether the latest session note is current enough to explain recent git changes and ongoing dirty work.

3. Inspect generated aids

Make sure hot memory and the scorecard are fresh enough to help, not mislead.

4. Check authority surfaces

Look for the boundaries that keep docs legible: docs/README.md, docs/archive/, and docs/drafts/.

5. Confirm the visual path

Ensure explainers have a stable home in .agent/diagrams/ so comprehension has a first-class surface.

Checks

Project truth checks

  • Baseline note exists.
  • Local source paths still resolve.
  • Latest session note tracks current work.
  • Dirty worktree outside memory is visible.

Reading path checks

  • Hot memory and scorecard are fresh enough to trust.
  • Docs authority boundaries are explicit.
  • Roadmap-like docs do not compete silently at docs root.
  • Visual explainers still have a stable home.
Output

PASS

Healthy enough for normal work. The surface still points at live project truth.

WARN

Usable, but drifting. This is where confusion starts if nobody tightens the spine.

FAIL

A required memory surface or trusted source path is broken. Fix this before pretending the repo is legible.

Real value

For the builder

  • Catches drift while the project is still fresh in your head.
  • Turns a vague "this feels messy" into explicit next actions.
  • Protects against phases being "done" in code but stale in docs.

For the future operator

  • Keeps the reading path calm and explicit.
  • Reduces the chance of opening the wrong doc first.
  • Improves handoff confidence before the original builder is absent.

For the agent

  • Improves restart speed without trusting stale summaries.
  • Keeps the repo legible under long execution threads.
  • Makes context retrieval safer because the sources stay curated.

The doctor is not the whole system. It is the moment where Context Spine stops being only a memory layer and starts acting like a lightweight project-truth maintenance system.
