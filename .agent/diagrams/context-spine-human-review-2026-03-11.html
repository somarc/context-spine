Context Spine Review Surface
As of 2026-03-11 · Humans first · Agents optional
Durable project understanding

Make the repo easier to restart, hand off, and trust.

Context Spine is not a shrine to one operator's memory. It is a small operating layer that keeps code, docs, notes, evidence, and retrieval close enough together that another person can pick up the project without a long oral download.

Design stance

The center of gravity is the project, not the agent. Live code still defines behavior. Docs still define intended behavior. Tests and command output still define what is trusted. Context Spine only makes those surfaces easier to recover and connect.

  • People can use it without Codex.
  • Teams can use it for onboarding and handoffs.
  • Agents benefit because the project is already legible.

Why it exists

Replace implied knowledge with visible surfaces.

Most project drift comes from three failure modes: the important mental model lives in one person, the evidence for decisions gets scattered, and restarts begin with rediscovery instead of retrieval. Context Spine targets those failures directly.

Restart friction

A developer returns after a week and has to reconstruct what is current, what was verified, and where the real truth lives.

Handoff friction

A teammate gets a verbal summary instead of a durable baseline note, session trail, and evidence-backed next action.

Search friction

The useful notes exist, but retrieval is stale or the paths are unclear, so the repo still feels opaque under time pressure.

Operating model

A five-layer stack with two loops.

The stack below is the whole point. Context Spine is valuable only when each layer sharpens the next one instead of pretending to replace it.

Layer 1 · Source of truth

Code, tests, logs, and commands

Delivered behavior lives here. Everything else should point back to this layer when certainty matters.

Layer 2 · Repo-local memory

Baseline notes, sessions, observations, evidence packs

Short-horizon continuity stays close to the repo so restarts and handoffs can begin from something real.

Layer 3 · Retrieval

QMD index over the right folders

Search should return the current baseline, the current session, and the current runbooks before broad repo scanning begins.

Layer 4 · Durable external knowledge

Linked long-horizon notes outside the repo

Deep dives, hub notes, and cross-project synthesis live here when they should outlast one codebase snapshot.

Layer 5 · Optional agent layer

Codex skill, local skills, and bounded delegation

Agents become more useful because the repo is better structured. They are not the foundation of the system.

Normal session

What a human actually does during meaningful work.

This is the practical loop the docs now describe in plain language. It works for a solo builder, a teammate inheriting work, or an agent-enabled workflow.

01

Bootstrap

Run the bootstrap entrypoint and let it surface the starting files instead of guessing where to begin.

02

Read baseline

Open the current spine-notes-*.md note to recover the project shape and working model.

03

Read session

Use the latest session summary if it exists. Create a new one when the task is substantial or the old note is stale.

04

Work from evidence

Change code, tests, docs, or scripts first. Avoid turning notes into speculative stand-ins for delivery.

05

Write back

Update the session note and log observations so another person can continue tomorrow without a recap call.

06

Refresh retrieval

Re-index notes and docs so future searches return the new truth instead of stale memory.

Use paths

Two explicit ways to operate it.

The docs no longer hide prerequisites. The wrapper path uses npm run. The direct path does not. Both now have exact commands and exact checks.

Wrapper path

For teams that want named repo commands

This is the README happy path. It requires node and npm because it uses package.json scripts.

git --version python3 --version bash --version node --version npm --version command -v qmd && qmd status npm run context:init npm run context:bootstrap npm run context:session
Direct path

For repos that want less wrapper machinery

This path keeps the same operating loop but uses the repo scripts directly. It does not require the npm wrappers.

git --version python3 --version bash --version command -v qmd && qmd status bash ./scripts/context-spine/init-qmd.sh bash ./scripts/context-spine/bootstrap.sh python3 ./scripts/context-spine/mem-session.py --project context-spine
Implemented here

What this repo now ships as the reviewed baseline.

The current implementation now tries to be explicit, installable, and reviewable. The optional Codex skill is intentionally placed as one card among many, not the center of the story.

Bootstrap

Start-here output is explicit

Bootstrap now surfaces the baseline note, recent visual explainer, latest session state, and quick-start commands.

Prerequisites

No more hidden wrapper assumptions

The README and prerequisites runbook now separate the wrapper path from the direct-script path with exact commands and cited sources.

Human docs

People-facing usage guide added

The repo now explains onboarding, handoffs, session notes, and team workflows without making the agent the main character.

Retention

Durable vs rolling vs generated is codified

Architecture and boundaries stay durable. Sessions and observations stay rolling. Hot-memory indexes and scorecards are treated as generated local artifacts.

Optional orbit

Codex skill shipped, not imposed

The repo owns a context-spine skill source and installer, but the project still makes sense without it.
