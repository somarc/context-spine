Context Spine · Native Memory Direction · ADR 0006

Make memory native without making it opaque.

Context Spine already has the right instincts for a Codex memory surface: explicit truth, durable structure, repo-local inspectability, and evidence-backed understanding. The next step is not to turn it into a control plane. The next step is to add a stronger machine layer around the same human-readable core.

What stays true

Files remain the project-truth layer. Native memory should index, accelerate, and reconcile those surfaces, not replace them with hidden state.

Human contract

The memory can be native and agent-centric internally. The user-facing requirement is simpler: Codex should visually explain what it knows, why it believes it, and what is uncertain.

Interface Shift

The memory is for Codex. The explanation is for the human.

That changes the target shape. Native memory does not need to expose every internal record as a file if it can render a trustworthy visual explanation with evidence and source-of-truth references.

Good / Bad / Ugly

Why Context Spine is a strong prototype, but not the final native form

Good

  • Explicit truth model across code, docs, tests, and evidence
  • Repo-local inspectability
  • Durable vs rolling vs generated memory separation
  • Skills as interpreters, not as the memory itself

Bad

  • Too much normal-use discipline still depends on people remembering the loop
  • Too file- and wrapper-shaped for a native runtime surface
  • Mostly scoped to one repo at a time
  • Structured schemas are still thinner than a native system needs

Ugly

  • It could turn into ritual instead of utility
  • It could become polished confidence theater
  • It could harden one operator style into a universal doctrine
  • It could quietly drift toward prompt soup or control-plane behavior
Layered Memory

Universal usefulness depends on explicit memory layers

Native-quality memory cannot stop at one repo. It needs bounded layers that can be queried independently and connected deliberately.

Layer 1

Session memory

  • current task
  • branch and git state
  • recent commands and tests
  • suspect assumptions and blockers
↔
Layer 2

Project memory

  • baseline notes
  • ADRs and runbooks
  • evidence packs and explainers
  • source-of-truth references
Layer 3

Operator / organization memory

  • cross-repo heuristics
  • shared playbooks
  • portable durable notes
  • domain overlays such as AEM and EDS
Boundary

Project memory stays repo-local by default

Broader memory should attach through adapters. The repo must not be polluted by hidden global state just because a native runtime exists.

Capture Loop

Reduce ceremony by capturing better evidence automatically

Input

Git + diffs

Branch state, changed files, and meaningful deltas should be available without manual copying.

Input

Commands + tests

Verification events should flow into memory with provenance, not just disappear into terminal history.

Input

Tool events

Edits, runs, and agent actions should produce structured records where they materially change understanding.

Machine layer

Evidence + session records

Native memory should store structured capture records that support retrieval and promotion.

Human layer

Durable truth

Only curated understanding gets promoted into baselines, runbooks, ADRs, and explainers.

Dual Surface

Keep human-readable project truth and add machine-usable structure around it

Human surfaces

What people reopen

  • markdown baselines
  • ADRs and runbooks
  • visual explainers
  • curated evidence packs
  • generated aids that guide reading but do not claim truth
Machine surfaces

What Codex queries and reconciles

  • session snapshots
  • evidence records
  • promotion records
  • artifact records
  • memory-layer and source-reference metadata
API Families

What a native interface would actually expose

capture

Record git state, diffs, commands, tests, and tool events with provenance.

query

Rank memory across session, project, and broader overlays without flattening them into one blob.

promote

Turn working knowledge into durable project truth with explicit evidence links.

invalidate

Mark stale or superseded assumptions explicitly instead of quietly leaving both stories live.

reconcile

Compare memory against code, docs, tests, and evidence to surface drift.

export

Make memory portable across repos, agents, and broader environments without losing inspectability.

Phases

How this can evolve without breaking the soul of the system

Phase 1

Better local memory discipline

  • stronger evidence and promotion schemas
  • more auto-capture from existing wrappers
  • safer generated aids
  • less manual upkeep in ordinary loops
Phase 2

First-class local memory APIs

  • local query and capture interfaces
  • explicit invalidation and supersession records
  • better ranking across session and project layers
  • structured exports
Phase 3

Layered memory beyond a single repo

  • user-level overlays
  • organizational adapters
  • cross-repo linking
  • portable memory references
Guardrails

What this direction must not turn into

Never become

  • a transcript archive
  • a control plane
  • prompt soup
  • a black-box memory product
  • confidence theater

Must stay

  • inspectable without a special runtime
  • clear about durable vs rolling vs generated memory
  • evidence-backed
  • generic at the core, domain-specific at the edge
  • simpler in daily use over time
