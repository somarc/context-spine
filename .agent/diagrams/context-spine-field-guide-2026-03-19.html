Project Memory Surface

Context Spine is the layer that makes a repo resumable again.

It is not a prompt archive and not an orchestration plane. It is a disciplined memory contract: keep the baseline, current thread, evidence, and reading path legible enough that a human, Codex, Claude Code, or the next maintainer can restart without paying the full rediscovery tax.

repo-local truth retrieval optional human + agent legibility explicit promote / invalidate
Design Stance

Keep the operating layer thin. Context Spine should compress recovery and handoff, not become another system the team has to babysit.

Truth flows upward. Runtime, code, tests, logs, docs, then durable notes. Retrieval helps discovery but does not replace the working set.

Humans and agents get different affordances. Agents need a bounded packet. Humans need a clear reading path and good visual explainers.

Frame

What it is, and what it refuses to become.

The system works when it stays boring in the right places: file-backed, inspectable, evidence-linked, and conservative about automation.

Context Spine Is
  • A restart surface: baseline note, latest session, hot-memory, evidence, and source-of-truth references.
  • A reconciliation discipline: when project truth changes, durable memory gets promoted or invalidated explicitly.
  • A legibility layer for both people and coding agents.
  • A small runtime contract around bootstrap, query, rehydrate, doctor, score, verify, promote, and invalidate.
Context Spine Is Not
  • Not a substitute for code, tests, logs, or docs.
  • Not a daemon, worker bus, or hidden memory database.
  • Not a reason to keep every thought forever.
  • Not a retrieval-first system where QMD or embeddings become the identity of the project memory model.
The point is not “more notes.” The point is less rediscovery, less re-explaining, and less blind inference.
Mental Model

Think in layers, not in one giant memory blob.

The elegant way to use Context Spine is to separate evidence, durable memory, rolling memory, and adapters. Each layer has a different job.

1. Evidence

Live repo truth

What actually happened or currently exists. This layer outranks everything above it.

code tests commands logs runtime output docs that own behavior
2. Durable Memory

Project model

The stable reading path: what the repo is, what parts matter, and how the project should be conceptualized over time.

spine-notes-*.md ADR runbooks visual explainers curated evidence packs
3. Rolling Memory

Current thread

The short-horizon continuity layer that lets the next restart land in the middle of active work instead of rebuilding context from zero.

latest session observations hot-memory-index next actions
4. Adapters

Recovery helpers

Commands and retrieval surfaces that make the memory usable without changing what counts as truth.

bootstrap query rehydrate doctor score verify QMD skills
Surfaces

The real operating surfaces

These are the files and commands that actually carry the system. If they drift, Context Spine stops paying for itself.

Surface Why it exists Human first move Agent first move
meta/context-spine/spine-notes-*.md Baseline project model and durable reading path. Open it to understand the repo shape. Use it as the durable anchor in bootstrap and query.
meta/context-spine/sessions/ Current continuity and explicit next actions. Read the latest session before asking for a recap. Use it to infer active objective and immediate next action.
meta/context-spine/hot-memory-index.md Curated working set for right-now reopening. Open the recommended files first. Treat it as a compression layer, not as authority.
docs/adr/, docs/runbooks/, .agent/diagrams/ Durable boundaries, operating knowledge, and visual system shape. Use one explainer plus the key runbook. Use as supporting authority after code and current evidence.
context:query / context:rehydrate Machine-readable recovery packets. Usually not needed unless you want the runtime view. Best spawn-time packet for Codex / Claude / other coding agents.
context:promote / context:invalidate Explicit durable-memory reconciliation. Use when a big truth changed. Use after file changes, not instead of file changes.
QMD Discovery and retrieval acceleration. Use when corpus size makes manual reopening slow. Use after the working set is bounded, not before.
Normal Loop

How to actually operate it

The loop is intentionally simple. The system loses value when this becomes ceremony.

01

Bootstrap

Run context:setup once, then context:bootstrap on restart. Open the baseline, the latest session, and the hot-memory working set.

02

Work from evidence

Read code, tests, logs, commands, and owning docs before synthesizing. Memory is downstream of the repo, not a replacement for it.

03

Capture what matters

Update the session, add an observation if needed, and keep the newest facts connected to evidence instead of letting them live only in scrollback.

04

Verify and reconcile

Run context:doctor, context:score, or context:verify when the reading path or trust level changed. Promote or invalidate durable memory explicitly.

05

Refresh retrieval

Use context:refresh or context:update after docs and notes change. Use embedding only when it materially helps, not by reflex.

Agent Pattern

How to leverage it with Codex, Claude, or another coding agent

The agent contract should be stable even when the tool changes. Codex and Claude have different runtimes, but both benefit from the same memory discipline.

Agent gets
  • a smaller, higher-signal working set
  • explicit next actions instead of vague restart prose
  • fewer blind assumptions about repo shape
  • reconciliation commands for durable memory changes
Human gets
  • a readable baseline and latest session
  • one visual entrypoint when the system is complex
  • evidence trails instead of magical agent claims
  • a repo that can be resumed without oral history
Shared rule
  • truth still starts in code, tests, commands, and logs
  • retrieval stays supportive, not sovereign
  • promotion and invalidation stay explicit
  • the memory layer should stay thin
Weak Agent Pattern
  • Start with a broad search over the whole corpus.
  • Treat old notes as equal to current runtime evidence.
  • Let every subagent write its own durable memory.
  • Assume “latest” from retrieval instead of dated files or commands.
Strong Agent Pattern
  • Bootstrap into the bounded working set first.
  • Use context:query or context:rehydrate as the spawn-time packet.
  • Let the parent agent stay the single writer for durable memory.
  • Promote or invalidate after actual file changes and verification.
Codex / Claude Workflow
  • Bootstrap or rehydrate.
  • Open the baseline, latest session, hot-memory, and source-of-truth files.
  • Do the coding task from live evidence.
  • Use bounded subagents only for narrow retrieval, review, or isolated edits.
  • Write session and reconciliation outputs at the parent level.
Best Leverage
  • Use Context Spine to reduce restart cost, not to replace engineering judgment.
  • Use it hardest when a repo is old, multi-surface, frequently handed off, or agent-assisted.
  • Pair it with visual explainers when humans need the shape fast.
  • Keep the contract generic enough that changing agent runtime does not require rewriting the memory model.
Adoption Ladder

Use only as much as the repo can keep healthy

Context Spine becomes elegant when the amount of process matches the complexity of the project.

Level 1
Minimal

One baseline note, one session directory, one bootstrap command. Enough for solo restart quality.

Level 2
Team-ready

Add ADRs, runbooks, observations, and one visual explainer. Enough for handoffs and onboarding.

Level 3
Agent-ready

Add query, rehydrate, verify, promote, invalidate, and a stable skill surface. Enough for coding agents to use the repo without guessing.

Level 4
Fleet-ready

Add rollout and upgrade paths so several repos can stay current without bluntly overwriting local customizations.

Signals

How to know it is helping, and how to know it is drifting

A good memory surface makes the repo feel easier to re-enter. A bad one becomes another stale layer to distrust.

Healthy Signals
  • Restarts begin from the right files without debate.
  • People ask for fewer “what were we doing?” recaps.
  • Agent outputs cite concrete files, tests, and commands.
  • Durable notes evolve only when the project model really changes.
  • Visual explainers reduce explanation time instead of duplicating it.
Anti-Patterns
  • Sessions turn into transcript dumps.
  • Retrieval becomes the default even when the needed files are already obvious.
  • Every note is treated as equally authoritative.
  • The memory layer gets bigger, but restart quality does not improve.
  • Subagents write parallel truths that nobody reconciles.
Bottom Line

If you want one sentence to carry forward: Context Spine is the thin memory contract that sits between the repo’s live evidence and the human or agent trying to restart work.

Use it to make the working set obvious, the evidence trail inspectable, and the durable project model explicit. Use retrieval, subagents, and visual explainers in support of that contract, not as substitutes for it.
