Context Spine · Upgrade Path

Upgrade older installs without stomping the project’s own truth.

The upgrade path is additive-first. It detects what a target repo is missing, scaffolds only the low-risk surfaces, and leaves project-owned files in a merge-review lane instead of pretending a boilerplate sync is safe.

Command

python3 ./scripts/context-spine/upgrade.py --target /path/to/project

Writes a local report into the target repo so the upgrade is explicit before anything changes.

Flow

1. Detect the install shape

Classify the target repo as missing, partial, or existing so the upgrade starts from the real state, not assumptions.

2. Separate safe additive files from merge-worthy files

Missing docs-governance surfaces and the doctor runbook are safe to add. Bootstrap, package scripts, and baseline wiring stay in a manual review lane.

3. Write an upgrade report

The report becomes the shared artifact for the upgrade decision, rather than a hazy “we probably updated Context Spine.”

4. Run doctor afterward

Once the upgrade lands, context:doctor checks whether the target repo now has a clean reading path.

Two Lanes

Safe additive

  • docs/README.md
  • docs/archive/README.md
  • docs/drafts/README.md
  • docs/runbooks/doctor.md

These can be created if missing without taking authority away from the target project.

Merge review

  • package.json
  • .gitignore
  • scripts/context-spine/bootstrap.sh
  • docs/runbooks/session-start.md
  • meta/context-spine/spine-notes-context-spine.md

These often carry repo-specific truth. The upgrade path calls them out instead of overwriting them.

Why this is worth it

For the maintainer

You can evolve Context Spine across projects without turning upgrades into risky manual archaeology.

For the project

The repo gets the newer hygiene surfaces without losing its custom baseline, commands, or memory voice.

For the future

The upgrade report becomes the durable record of what changed and what still needs judgment.
