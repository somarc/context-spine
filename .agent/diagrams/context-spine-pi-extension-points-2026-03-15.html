Context Spine · PI Extension Points

`.pi/` is the optional agent layer, not the project’s memory core.

Use .pi/ when a repo needs project-owned agent workflows that travel with the code. Keep architecture, status, runbooks, and continuity out of it. The point is to extend the memory system, not replace it.

Current stance

.pi/skills/ for named workflows. .pi/prompts/ for reusable scaffolds. Everything else stays in the normal code, docs, and memory surfaces.

Where it fits

Code

Delivered behavior and scripts that actually execute.

Docs

ADRs, runbooks, and durable explanations.

Memory

Baseline notes, sessions, observations, and evidence.

`.pi/`

Optional project-owned agent extensions that point back into the real truth surfaces.

What belongs there

Skills

  • named workflows
  • review lenses
  • bounded investigations
  • domain-specific procedures

Prompts

  • reusable scaffolds
  • handoff starters
  • brief templates
  • delegation output contracts
What does not belong there

Architecture truth

Use docs/adr/, not agent-only files, for system decisions.

Current status

Use meta/context-spine/ for continuity instead of hiding status in prompts.

Runtime logic

Put real behavior in normal scripts or product code, not in prompt scaffolding.

Decision rule

Use `.pi/skills/` when the workflow deserves a name

It recurs, benefits from a stable procedure, and should be installable into an agent runtime.

Use `.pi/prompts/` when the framing matters more than automation

The work needs a repeatable prompt shape, but not a full skill or script.

Use normal docs or scripts for everything else

If it is project truth, operational procedure, or executable logic, it should live outside .pi/.
